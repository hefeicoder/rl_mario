How to read this Eight chapters, each solving the problem the previous one created. We start from the bare RL loop, reach PPO's clipped objective by Chapter 5 (the summit), then put it to work and learn to read and rescue real training runs. 1
The problem
Mario as an RL loop

A person sees Super Mario Bros and just… plays. An algorithm sees a wall of 184,320 pixel numbers (240×256×3) that change every frame, and a single number — a reward — trickling back. No rulebook, no labels, no "press jump here." Reinforcement learning is the craft of turning that trickle of reward into skilled behavior. This chapter sets up the problem; the rest of the document builds the algorithm that solves it.

The loop

All of RL is one loop. An agent and an environment take turns: the agent picks an action, the environment reacts and hands back a new observation and a reward, forever (or until the episode ends).

        action a_t
   ┌───────────────────┐
   │                   ▼
┌──┴───────┐    ┌───────────────┐
│  AGENT   │    │  ENVIRONMENT  │
│ (policy) │    │  (the game)   │
└──────────┘    └───────┬───────┘
   ▲                    │
   └────────────────────┘
   observation s_t+1, reward r_t

The agent–environment loop. Everything in this document is about making the box on the left better.

The five pieces

  • State / observation — what the agent sees right now (the screen).
  • Action — what it can do (a button press).
  • Reward — a single number scoring the last action. The only feedback the agent ever gets.
  • Policy — the agent's strategy: a function $\pi(a \mid s)$ giving the probability of each action in a state. This is the thing we train.
  • Return — the total reward over a whole run, with future reward discounted by $\gamma$ (gamma):
$$ R = r_0 + \gamma r_1 + \gamma^2 r_2 + \cdots = \sum_{t=0}^{\infty} \gamma^t\, r_t $$

The discount $\gamma \in [0,1)$ makes near-term reward worth more than far-future reward — partly because the future is uncertain, partly to keep the sum finite. The agent's entire goal is to pick actions that maximize the expected return $R$.

In our Mario run Every piece is concrete in this project. State: 4 stacked grayscale 84×84 frames (4 frames so motion is visible). Actions: the 7 SIMPLE_MOVEMENT moves (right, right+jump, …) instead of all 256 NES button combos. Reward: move-right progress, minus a small time penalty, minus a death penalty — so "good" literally means move right, stay alive, reach the flag. We read progress off x_pos (the flagpole sits near 3160). Discount: $\gamma = 0.9$.

The question that drives everything

We have a loop, a policy, and a reward. The whole game is: given only the rewards we observe, how do we change the policy so it earns more of them? The next chapter gives the foundational answer — and every chapter after it fixes a problem the previous answer created, until we arrive at PPO.

2
The foundational answer
Policy gradients: push up what worked

Chapter 1 left us with a question: given only rewards, how do we make the policy better? Here is the oldest, simplest, most honest answer — the idea every modern method (PPO included) is built on top of. It has a name, REINFORCE, but the idea is what matters.

The core idea, in one sentence

Play an episode. Then make the actions that led to high return more likely, and the actions that led to low return less likely. That's it. If a run went well, do more of what you did; if it went badly, do less. Repeat a few million times and the policy drifts toward behavior that earns reward.

   one episode of Mario:

   s0 → a0:RIGHT      → return was HIGH  →  ↑ make RIGHT here more likely
   s1 → a1:RIGHT+JUMP → return was HIGH  →  ↑ make this jump more likely
   s2 → a2:LEFT       → return was LOW   →  ↓ make LEFT here less likely
   s3 → a3:DOWN       → return was LOW   →  ↓ make DOWN here less likely

The whole algorithm: nudge each action's probability up or down by how good the run was.

The one equation (and how to read it)

Written as a gradient — the direction to nudge the policy's weights $\theta$ to increase expected return — it's just that sentence in symbols:

$$ \nabla_\theta J(\theta) = \mathbb{E}\big[\, \nabla_\theta \log \pi_\theta(a \mid s)\; \cdot\; R \,\big] $$

Read it in two halves:

  • $\nabla_\theta \log \pi_\theta(a \mid s)$ — "the direction in weight-space that makes this action more likely." Pure book-keeping from the network.
  • $\cdot\, R$ — "scaled by how good the run was." A big positive return pushes hard in that direction; a negative return pushes the opposite way.

Multiply them, average over experience, take the step. No labels, no "correct action" — the reward alone decides which way each action's probability moves. That's the magic of policy gradients.

The catch: it's incredibly noisy

Here's the problem that the next three chapters exist to fix. The return $R$ for a whole episode is a single, wildly variable number. Mario might play almost identically twice and die to a random Goomba once and survive the other time — same actions, very different $R$. So every action in the episode gets credited or blamed for the luck of the whole run. The gradient estimate is correct on average, but so noisy that learning crawls.

Where we're headed Chapter 3 attacks this noise (better credit assignment). Chapter 4 attacks a second problem — updates that are too big. Chapter 5 is PPO, which solves the second problem with one clean trick.
In our Mario run This noise is exactly why a single episode's max_x_pos bounces around so much, and why we watch a smoothed curve over ~100 episodes rather than reacting to any one run. The learning signal is real, but it's buried in luck — and reducing that noise is most of what makes PPO trainable.
3
Fixing problem #1: noise
Baselines, value functions & advantage

Policy gradients work, but credit every action with the luck of the whole episode — so the signal is drowned in noise. The fix is a single shift in perspective that powers every modern actor-critic method.

Don't ask "was it good?" Ask "was it better than expected?"

Suppose moving right always earns about +30 reward. An action that earns +30 isn't good — it's average. An action that earns +50 is what you want more of; one that earns +10 is what you want less of. So instead of scaling by the raw return $R$, subtract a baseline — "what we expected from here" — and scale by the difference. Same direction of learning, a fraction of the noise, because the big common chunk of reward cancels out.

Value and advantage

The natural baseline is the value function $V(s)$ = "the return I expect from state $s$ if I play normally." Subtracting it gives the advantage:

$$ A(s,a) = R - V(s) \quad\;\; \begin{cases} > 0 & \text{better than expected} \rightarrow \text{push } \uparrow \\ < 0 & \text{worse than expected} \rightarrow \text{push } \downarrow \end{cases} $$

Now the policy-gradient step uses $A$ in place of $R$. Same idea as Chapter 2 — make good actions likelier — but "good" now means "beat the baseline," which is a far calmer, more informative signal.

   Chapter 2 (raw return R):      Chapter 3 (advantage A):
   a0  ↑↑↑↑↑↑↑  (R=+52)         a0  ↑↑       (A=+22, a bit above avg)
   a1  ↑↑↑↑↑↑↑  (R=+48)         a1  ↓        (A=–2,  a bit below avg)
   a2  ↑↑↑↑↑↑   (R=+44)         a2  ↓↓↓      (A=–26, clearly bad)
   big noisy pushes, all up        small honest pushes, signed

Subtracting the baseline turns "everything was positive, who knows which mattered" into a clean better/worse signal.

Actor-critic, and the GAE knob

But where does $V(s)$ come from? We learn it too, with a second network (or a second head). This is the actor-critic setup: the actor is the policy $\pi$, the critic is the value estimate $V$. The critic judges; the actor improves based on the judgement.

One more refinement you'll see named in configs: GAE (generalized advantage estimation). It's a dial, $\lambda$, for how you estimate the advantage — trading off bias against variance. Small $\lambda$ = lower variance but more biased; large $\lambda$ = the reverse. You rarely touch it; just know it's the knob behind a cleaner advantage signal.

In our Mario run The TensorBoard metric train/explained_variance (~0.86 in our run) is literally a report card for the critic: how much of the real return its $V(s)$ predictions account for. High means the baseline is trustworthy, so the advantage — and every policy update built on it — is clean. We use gae_lambda = 0.95.
4
Fixing problem #2: instability
The other catch: updates that blow up

Chapter 3 cleaned up the direction we push the policy. But there's a second, sneakier problem: how far we push. Take too big a step and training doesn't just slow down — it can detonate.

Why one greedy step can destroy a good policy

Here's the trap that makes RL different from ordinary supervised learning. We collected our experience using the current policy. The gradient says "this direction looks good" — but it's only trustworthy near the current policy. Take a giant step and you land at a policy so different that the data you used no longer describes it at all. The move that looked like +reward on paper can collapse behavior entirely. And because the next batch of data is collected by this broken policy, there's nothing to pull it back. It can spiral.

   surrogate reward
        ▲
        │        .-•-.          ← small step: climb toward the peak
        │      .'     '.
        │    .'         '•        ← the model THINKS reward keeps rising...
        │   /             \
        │  •               \________
        │                          ↓
        │                      (real policy falls off a cliff)
        └────────────────────────→  how far we moved the policy

The surrogate objective is only accurate near the old policy. Trust it too far and you walk off a cliff.

The fix: stay in a trust region

The cure is intuitive: don't let the policy change too much in a single update. Keep each step inside a "trust region" where the data still roughly describes the policy. The classic rigorous version is TRPO (Trust Region Policy Optimization). It measures how different the new policy is from the old using KL divergence — think of it as a "distance between two policies" — and mathematically constrains each update so that distance stays small.

TRPO works and was a milestone. The downside: enforcing that constraint exactly involves heavy machinery (second-order optimization) that's fiddly to implement and slow to run. Which sets up the obvious question — the one PPO answers: could we get this same "don't move too far" safety with something far simpler?

In our Mario run The metric train/approx_kl (~0.01 when healthy) is exactly this "how far did the policy move this update" measure. We watch it as a stability gauge: a sudden spike means an update was too aggressive. PPO, next chapter, is what keeps it small without TRPO's complexity.
5
The summit
PPO: the clip that changed everything

Everything so far has been a setup. We have a clean learning direction (advantage, Ch 3) and a clear danger (steps that are too big, Ch 4). PPO is the idea that earns "don't move too far" with a single, almost embarrassingly simple change to the objective — no second-order math, no hard constraint. This is the whole point of the document.

First, the probability ratio

To talk about "how much the policy changed," PPO looks at one number per action — the ratio of how likely the new policy is to take that action versus the old one:

$$ r(\theta) = \frac{\pi_\theta(a \mid s)}{\pi_{\theta_{\text{old}}}(a \mid s)} $$

$r = 1$ means "unchanged." $r = 1.3$ means the new policy is 30% more likely to take this action; $r = 0.7$ means 30% less. The ratio is the "how far did we move" signal, measured action by action.

The clipped objective

Naively, we'd maximize $r(\theta)\,A$ — push the ratio up for good-advantage actions. But that's exactly what walks off the cliff: nothing stops $r$ from ballooning. PPO's fix is to clip the ratio and keep the worse of the two:

$$ L(\theta) = \mathbb{E}\Big[\, \min\big(\, r(\theta)\,A,\;\; \operatorname{clip}(r(\theta),\, 1-\epsilon,\, 1+\epsilon)\,A \,\big) \Big] $$

That $\operatorname{clip}$ just means "don't let $r$ leave the band $[1-\epsilon,\, 1+\epsilon]$." With $\epsilon = 0.1$, the policy is never rewarded for moving any single action's probability more than 10% per update. The trust region — for free, with plain gradient descent.

Why it works — a picture, not a proof

The $\min$ looks fiddly until you draw it. Think about one action with a good advantage ($A > 0$), and ask how the objective rewards moving its probability:

   objective
   (A > 0)
        ▲
        │            ____________   ← past 1+ε: FLAT.
        │           /                  no more reward for
        │          /                   pushing this action.
        │_________/
        │        |
        ────────┼─────────────→ ratio r
              1   1+ε

   good action: reward rises as you make it likelier, then STOPS at 1+ε.
   bad action (A < 0): mirror image — no reward for shoving it past 1–ε.

The clip flattens the objective once the policy has moved "far enough," removing any incentive to overshoot. That flat line is the entire trick.

So a good action gets more likely — but only up to a point, then the gradient goes to zero and the update stops pushing. A bad action gets less likely, with the same brake in the other direction. No constraint solver, no KL penalty to tune — just min and clip. That's why PPO took over: TRPO's stability at a fraction of the complexity, and it plays nicely with ordinary minibatch SGD.

In our Mario run You can watch the clip fire: train/clip_fraction (~0.3 in our run) is the fraction of sampled actions whose ratio hit the $1\pm\epsilon$ band and got clipped. We set $\epsilon$ via clip_range = 0.1 in src/config.py. A healthy clip_fraction means the brake is engaging on the right number of updates — not too loose, not strangling learning.
6
From idea to algorithm
PPO in practice: the full loop

We have all the pieces. Here's how they snap together into the loop that actually runs when you type model.learn() — and what every hyperparameter in our config is really controlling.

The full PPO loop

PPO alternates between collecting fresh experience and learning from it, over and over:

   repeat forever:

   1. COLLECT   run the current policy in N parallel envs for
                a fixed number of steps  →  a batch of rollouts

   2. ESTIMATE  compute advantages A for every step (GAE, Ch 3)

   3. UPDATE    for K epochs, in minibatches:
                  • compute the ratio r (Ch 5)
                  • take a gradient step on the CLIPPED objective
                  • (the clip keeps each step in the trust region)

   4. DISCARD   throw the batch away, go back to step 1 with the
                improved policy

Collect → estimate → clipped update → repeat. PPO is "on-policy": each batch is collected by the latest policy and used only briefly, which is why steps 1 and 4 bookend every update.

The knobs (and what they do)

Now PPO's hyperparameters read as plain English. These are our actual values from src/config.py:

  • n_steps = 512 — how many steps each env collects before an update (rollout length).
  • N_ENVS = 8 — how many emulators run in parallel for diverse, decorrelated experience.
  • n_epochs = 4 — how many times we reuse each batch before discarding it (the clip is what makes this safe).
  • batch_size = 256 — minibatch size for each gradient step.
  • clip_range = 0.1 — $\epsilon$, the trust-region width from Chapter 5.
  • gamma = 0.9 — the discount / horizon (Ch 1).
  • gae_lambda = 0.95 — the bias/variance dial on the advantage (Ch 3).
  • ent_coef = 0.01 — an entropy bonus that rewards keeping the policy a little random, so it doesn't stop exploring too soon. (This one becomes the hero of Chapter 8.)

The same thing, in code

In this project it's a few lines of Stable-Baselines3 — the library implements the loop above; we just hand it the env and the knobs:

from stable_baselines3 import PPO

model = PPO("CnnPolicy", venv, n_steps=512, n_epochs=4,
            clip_range=0.1, gamma=0.9, gae_lambda=0.95,
            ent_coef=0.01, ...)
model.learn(total_timesteps=4_000_000)   # runs collect→estimate→update
In our Mario run Those 8 parallel envs are why we measure ~207 env-steps/s on an Apple M1 Max, and why training is emulator-bound on the CPU, not GPU-bound — the bottleneck is stepping 8 NES emulators, not the tiny CNN. It's also why a 4-million-step run takes ~5.4 hours here.

Hero skill #1
Reading a healthy run

Anyone can call model.learn(). The skill that separates "ran PPO" from "understands PPO" is reading the curves while it trains — knowing what healthy looks like, so you can tell when it isn't. Here's how we read our Mario runs in TensorBoard.

The metrics that matter, in three groups

  • Progress — mario/flag_rate (our win rate; pinned at 0 until the agent can run most of the level) and mario/max_x_pos (how far right it gets, climbing toward the flagpole at ~3160).
  • Behavior — rollout/ep_rew_mean (overall reward) and rollout/ep_len_mean (episode length). A lovely tell: episode length rising then falling means the agent stopped dawdling to survive and started running — covering more ground in fewer steps.
  • Health — the diagnostics from earlier chapters: explained_variance (~0.86, the critic is good, Ch 3), approx_kl (~0.01, updates aren't too big, Ch 4), clip_fraction (~0.3, the clip is engaging, Ch 5), and entropy slowly decaying (the policy growing more confident).

The shape of learning: bursts, not a smooth line

New practitioners expect a tidy upward curve. Real RL looks like this:

   max_x_pos
   3160 │- - - - - - - - - - - - - - - - - - - - - flag
        │
        │                                    .•´  ← breakthrough!
        │                              _.•´
        │        plateau          _.•´             ← cleared the
        │      .•—•—•—•—•—•—•.•´                    obstacle, easy
        │    .´       (stuck ~800)               ground unlocked
        │  .´
        │.´  fast early gains
        └────────────────────────────→ steps

Climb → plateau at an obstacle → breakthrough. Progress arrives in bursts because clearing one hard spot unlocks a stretch of easy level.

And it's noisy and non-monotonic — the curve dips and recovers constantly. Two reasons you already know: max_x_pos is a rolling average over recent episodes (Ch 1), and the policy samples its actions (Ch 2), so individual episodes vary. Watch the smoothed trend, never a single step. A dip that recovers is normal; only a curve that climbs then collapses and stays down (with approx_kl spiking) is real instability.

One eval gotcha: stochastic vs deterministic

During training PPO samples actions (it must explore). When you watch a trained agent, you can either keep sampling or take the single most-likely action (deterministic, argmax). For a strong agent, deterministic is crisp and optimal. For a weak agent on a deterministic emulator, argmax is brittle — it plays the exact same doomed run every time, while sampling at least explores. Rule of thumb: sample while learning, go deterministic for the final winner.

In our Mario run We added mario/flag_rate and mario/max_x_pos ourselves (a few logger.record(...) lines) precisely so the dashboard shows progress toward winning, not just reward going up. The win rate sitting at 0 while max_x_pos climbs is the expected early picture — the flag only gets reached once the agent can run nearly the whole level.
8
Hero skill #2
When training gets stuck

This is the chapter that separates "I ran PPO once" from "I can get PPO to work." Sooner or later your agent stops improving — the curve goes flat. The real skill is knowing whether to wait or intervene, and which lever to pull. We learned this one the hard way on Mario, so the example at the end is the real thing.

First question: stuck, or just slow?

A flat curve is not automatically a problem — RL plateaus constantly (Chapter 7). Before touching anything, tell the two apart:

  • Just slow (wait): entropy is still healthy (the policy hasn't gone fully deterministic), max_x_pos occasionally poke upward, and the agent is clearly still trying things. It's grinding through a hard section.
  • Genuinely stuck (intervene): entropy has collapsed toward zero, the curve is dead flat for a long stretch (hundreds of thousands of steps), and approx_kl is tiny — the policy has stopped changing. It has settled into a local optimum and won't leave on its own.

Then diagnose before you tune. The single most useful move: watch the agent play and see where it dies — a specific pipe? a pit it won't jump? That tells you whether you have an exploration problem or something else, far better than staring at curves.

The escape levers, in order

Reach for these roughly top to bottom — cheapest and most common first:

  1. Patience / more steps. The most underrated fix. Many "stuck" runs just needed longer — see the example below.
  2. Exploration (ent_coef). Raise the entropy bonus so the policy keeps trying new actions and can stumble onto the escape. The first real knob for a genuine plateau.
  3. Horizon (gamma). Raise it (e.g. 0.9 → 0.99) when success needs valuing a delayed payoff — a running jump that sacrifices immediate progress for reward on the far side.
  4. Learning rate / clip_range. Lower them if training is unstable (climbs then collapses) rather than merely stuck.
  5. Reward shaping. Add your own bonuses to point the agent at what matters, when the built-in reward is too sparse.
  6. Curriculum. Start on an easier version of the task and ramp difficulty — the heaviest lever, for genuinely hard problems.

A decision flowchart

   curve went flat
        │
        ▼
   entropy collapsed?  ──no──►  probably just SLOW → wait / more steps
        │ yes
        ▼
   watch it play: where does it die?
        │
        ├─ stuck at an obstacle, never varies  ──►  raise ent_coef (explore)
        ├─ needs a risky delayed-payoff move   ──►  raise gamma (horizon)
        ├─ climbs then COLLAPSES, kl spikes    ──►  lower lr / clip_range
        └─ reward too sparse to guide it        ──►  reward shaping / curriculum

Stuck? → check entropy → watch where it dies → pick the matching lever.

Worked example: two real runs, two outcomes

Here is the lesson in one picture from this project. Two PPO runs on World 1-1, both hitting the same wall around x_pos ≈ 800–900:

   max_x_pos
   1600 │                                   _.•´   PPO_3
        │                              _.•´´       (broke out!)
   1200 │                         _.•´
        │                    _.•´
    900 │   .•—•—•—•—•—••´——•—•—•—•—•—•—•  baseline (stuck ~895)
        │ .´        ↑              ↑
        │.´     both plateau     PPO_3 breaks out ~320k,
        └───────────────────────────→ steps
        0       100k    200k    300k    400k    500k

Same wall, opposite endings. The contrast is the whole lesson.

PPO_3 sat at ~800–850 from 100k to 300k steps — the exact plateau that looks like failure — then broke out on its own around 320k and climbed to ~1,610 by 598k. No tuning. It just needed patience (lever #1). Meanwhile ppo_1M_baseline_1 hit the same wall and stayed flat at ~895 — a genuinely stuck run that would have needed intervention.

That's the judgment this chapter is teaching: a plateau is not a verdict. Check entropy, watch where it dies, and decide — sometimes the right move is to wait, and sometimes it's to reach for a lever. Knowing which is the difference.
